# Context Compiler — Скилл сбора контекста

Автономный агент, который сканирует кодовую базу проекта и опрашивает NotebookLM,
создавая детализированный `.ai/CONTEXT.md` — фундамент для любой сложной задачи.

## Когда использовать

- **Перед сложной задачей** (>2 файлов, незнакомый модуль)
- **При первом входе в проект** — нужно понять стек, паттерны, зависимости
- **Когда пользователь просит**: "собери контекст", "проанализируй проект", "context"
- **Перед архитектурными решениями** — нужна полная карта проекта
- **При работе в режиме `hard`** — обязательный первый шаг

## Когда НЕ использовать

- Простые правки в 1-2 файла (fast-режим)
- Задача уже знакома и `.ai/CONTEXT.md` актуален
- Нет notebook_id для проекта в NotebookLM

## Как вызвать

Используй MCP-инструмент `build_context` из сервера `context-compiler`:

```
call_mcp_tool(
  ServerName="context-compiler",
  ToolName="build_context",
  Arguments={
    "task": "Описание задачи",
    "notebook_id": "ID ноутбука NotebookLM",
    "workspace": "/путь/к/проекту"
  }
)
```

### Параметры

| Параметр | Обязательный | Описание |
|----------|:---:|-----------|
| `task` | ✅ | Описание задачи. Чем конкретнее — тем точнее контекст |
| `notebook_id` | ✅ | ID ноутбука NotebookLM с документацией проекта |
| `workspace` | ❌ | Путь к проекту (по умолчанию: `/Users/seva/Projects/saas-platform`) |

### Как узнать notebook_id

1. Вызови `notebook_list` через MCP `notebooklm-mcp`
2. Найди ноутбук проекта по имени
3. Возьми его `id`

## Что происходит под капотом

1. **Bootstrap** — сканирует структуру проекта (4 уровня), читает конфиги
2. **Декомпозиция** — определяет затронутые домены (API, DB, UI, и т.д.)
3. **Сбор** — задаёт 25+ целевых вопросов NotebookLM по 7 категориям, верифицирует каждый ответ кодом
4. **Самоаудит** — оценивает уверенность по 7 аспектам, досбирает пробелы
5. **Компиляция** — формирует структурированный CONTEXT.md с источниками

**Время работы:** 3-10 минут (зависит от размера проекта и загрузки API)

## Что получается на выходе

Файл `.ai/CONTEXT.md` со следующими секциями:

- **Стек проекта** — технологии с версиями и SOURCE
- **Что реализовано** — детально по каждому модулю с путями к файлам
- **Паттерны** — как принято в проекте (с примерами кода!)
- **Зависимости** — полный список с версиями
- **НЕ ДЕЛАТЬ** — антипаттерны и ограничения
- **Интеграции** — карта связей между компонентами
- **Тестирование** — фреймворк, моки, фикстуры
- **Env переменные** — релевантные задаче
- **Уверенность** — самооценка по 7 аспектам

Также сохраняется копия в `.ai/context_history/CONTEXT_YYYYMMDD_HHMMSS.md`.

## Пример использования в воркфлоу

```
# 1. Пользователь даёт сложную задачу
"Реализовать систему уведомлений (email + in-app)"

# 2. Агент вызывает context-compiler
build_context(
  task="Реализовать систему уведомлений пользователей (email + in-app)",
  notebook_id="4169e0fe-4c22-4278-a37e-8fda46efbd55",
  workspace="/Users/seva/Projects/saas-platform"
)

# 3. Читает сгенерированный контекст
view_file(".ai/CONTEXT.md")

# 4. Приступает к реализации с полным пониманием проекта
```

## Требования

- MCP-сервер `context-compiler` должен быть запущен
- Env-переменная `GEMINI_API_KEY` настроена
- Проект загружен в NotebookLM (есть notebook_id)
- Модель: Gemma 4 31B-IT (1500 RPD на free tier)

## Связанные скиллы

- **notebooklm** — управление ноутбуками, добавление источников
- **architecture-map** — генерация карты проекта (после сбора контекста)
- **deep-context** — для задач требующих сверхглубоких рассуждений
