# changelog

## Context (что читать перед стартом)

```
git log --oneline --no-merges <since>..<until>  → raw data
.ai/PROJECT_MAP.md                               → scope проекта
.foryou/CHANGELOG.md                             → текущий файл (если есть)
```

## Procedure

1. **Определить диапазон** — спросить пользователя или взять автоматически:
   - Есть теги → от последнего тега до HEAD: `git log v<last>..HEAD --oneline --no-merges`
   - Нет тегов → последние 20 коммитов: `git log -20 --oneline --no-merges`
   - Пользователь указал диапазон → использовать буквально

2. **Собрать коммиты** — запустить команду, получить список. Если коммитов 0 → сообщить «нечего записывать» и выйти.

3. **Сгруппировать по типам** (из conventional commits):
   ```
   feat     → ✨ Новые возможности
   fix      → 🐛 Исправления
   perf     → ⚡ Производительность
   refactor → ♻️ Рефакторинг
   docs     → 📝 Документация
   test     → 🧪 Тесты
   chore    → 🔧 Служебное
   ci       → 🤖 CI/CD
   build    → 📦 Сборка
   ```
   Коммит без префикса → секция `🗒️ Прочее`. Merge-коммиты → пропускать.

4. **Определить версию**:
   - Есть `package.json` → взять `version`
   - Нет → использовать дату: `YYYY-MM-DD`
   - Пользователь назвал версию → использовать её

5. **Написать секцию** в формате Keep a Changelog (https://keepachangelog.com):
   ```markdown
   ## [<version>] — <YYYY-MM-DD>

   ### ✨ Новые возможности
   - <scope>: <описание на русском> (`<hash>`)

   ### 🐛 Исправления
   - <описание> (`<hash>`)
   ```
   <!-- Few-shot правило Карпаты: конкретный шаблон с примером лучше абстрактного описания -->

   Правила описания:
   - Писать что изменилось ДЛЯ ПОЛЬЗОВАТЕЛЯ, не что поменялось в коде
   - Плохо: «рефакторинг функции validateEmail»
   - Хорошо: «валидация email теперь принимает unicode-домены»
   - Scope в скобках если важен контекст, иначе опустить
   - Хеш — короткий (7 символов), в backtick

6. **Обновить файл** `.foryou/CHANGELOG.md`:
   - Файл существует → вставить новую секцию ПОСЛЕ заголовка `# Changelog`, ПЕРЕД предыдущей версией
   - Файл не существует → создать с шапкой:
     ```markdown
     # Changelog

     Все значимые изменения проекта фиксируются здесь.
     Формат: [Keep a Changelog](https://keepachangelog.com/ru/)
     ```
   - NEVER перезаписывать старые секции

7. **Вывод пользователю**:
   ```
   ✅ CHANGELOG обновлён → .foryou/CHANGELOG.md
   Версия: <version> | Коммитов: <N> | Секций: <K>
   Пропущено: <M> merge-коммитов
   ```

→ After: предложить `git-commit` с сообщением `docs(changelog): добавить <version>`
